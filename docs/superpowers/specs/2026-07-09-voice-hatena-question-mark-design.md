# voice-hatena-question-mark 設計書

## 目的・スコープ

音声入力で「まる」→「。」のような日本語ディクテーションの定番コマンドと同様に、「はてな」と発話した際に文字通り出力せず「？」へ変換する。

- 対象: Handy のみ（`handy/ja_light_tidy.prompt.txt` の後処理プロンプト修正）
- 対象外: Typeless。理由は下記「対象外の判断根拠」を参照
- 対象外: 「まる」「てん」等、他のディクテーションコマンド語の追加（YAGNI。今回は「はてな」のみ）

## 背景（現状のHandyアーキテクチャ）

Handyは STT（音声→生テキスト）の出力を `ja_light_tidy.prompt.txt` というプロンプトとともに LLM（ローカルollama or Cerebras）に渡し、整形済みテキストを返す2段構成。既存のRule 1が「文脈から疑問文と判断した場合に？を自動付与する」処理を担っており、Rule 5が用語集（`glossary.txt`、「誤変換→正表記」形式）による固有名詞置換を担っている。

「はてな」を明示的に発話して？に変換する処理は、この2つのどちらとも性質が異なる：

- Rule 1（文脈判定）は「発話者が疑問符と言っていない場合でも文脈から自動判定する」ロジックであり、明示コマンドの処理を混ぜると責務が肥大化する
- Rule 5（用語集）は「音の近い誤変換やカタカナ表記」の1:1置換に限定される設計で、「はてな」は誤変換ではなく正しい語そのものであるため意味的に不整合。かつ用語集置換には位置制約がなく、「はてなブックマーク」等の複合語中の「はてな」まで無条件に変換されてしまう

よって、独立した新規ルールとして追加する。

## 対象外の判断根拠（Typeless）

TypelessのPersonal Dictionary機能をUIで直接確認した結果（「辞書」画面、「新しい単語を追加」ダイアログ）、入力欄は単一語のみで、既存エントリ（Claude, cowork, excel, notion 等）もすべて単一語。これはSTT認識精度向上のための固有名詞・語彙リストであり、「入力語を削除して別の記号に置換する」コマンド機能ではない。

公式ドキュメント（Help/FAQ/Quickstart/リリースノート）および第三者レビューを調査したが、Typelessに「発話中の特定語を記号に変換する」音声コマンド機能の記載は見つからなかった。存在するのは自動句読点（文脈判定、Handy Rule 1相当）とPersonal Dictionary（単語登録）のみ。

以上により、現バージョンのTypelessでは本機能を実装する手段がないと判断し、対象外とする。Issue側にこの判断根拠を記録し、Typelessが将来コマンド的な変換機能を追加した場合に再検討する。

## 設計

### 変更ファイル

`handy/ja_light_tidy.prompt.txt` のみ。`handy/glossary.txt` は変更しない（用語集の意味的スコープを守るため、はてな→？はここに含めない）。

### 新規ルールの追加

既存Rule 1（句読点ルール）の直後に新規Rule 2として挿入し、既存Rule 2〜9をRule 3〜10へ番号シフトする（内容は無変更）。

追加する文言:

> 2. 発話の実質的な文末（直後にSTTが付けた句読点のみが続く場合を含む）に「はてな」が単独で現れた場合、その「はてな」自体は出力せず、文末に？を1つだけ付ける（1の終端記号ルールより優先する）。「はてなマーク」「はてなブックマーク」等、他の文字・語と結合して名詞として使われている場合や文末以外に現れる場合は通常の語として扱い、変更しない。

### 整形例セクションへの追加

既存の「以下は整形の例です」ブロックに、正例・負例を1組追加する。

```
入力: 今日中に終わりますはてな
出力: 今日中に終わります？

入力: はてなブログに投稿しておきます
出力: はてなブログに投稿しておきます。
```

正例は「文末の明示コマンド」が文脈判定なしに？へ変換されることを示す。負例は複合語中の「はてな」が誤爆しないことを示す。

### 反映方法

コード変更は不要。`handy-switch <provider>` で再適用（既存の運用フローと同一）。

## スコープ外（YAGNI）

- Typeless対応（上記「対象外の判断根拠」参照。将来Typelessが対応機能を追加した場合の別Issueとして扱う）
- 「まる」「てん」「かいぎょう」等、他のディクテーションコマンド語の追加
- 「はてな」の複合語判定を厳密な形態素解析で行うこと（プロンプトの自然言語ルールで十分と判断。誤爆が実運用で頻発した場合に再検討）

## テスト方針

自動テストなし（既存のHandy運用と同様、手動トレース + 実機発話テスト）。

1. 変更後の `ja_light_tidy.prompt.txt` を目視で確認し、Rule番号のシフトが正しく行われていること、既存Rule 3〜10の内容が変更前のRule 2〜9と一致することを確認する
2. 紙上トレース: 正例「今日中に終わりますはてな」→「今日中に終わります？」、負例「はてなブログに投稿しておきます」→「はてなブログに投稿しておきます。」がルール文言から導出できることを確認する
3. `handy-switch` で実プロバイダ（local/cloud）に再適用し、実機で「（何か）はてな」と発話して？に変換されることを確認する
4. 実機で「はてなブックマークに登録しておきます」等、複合語を含む発話を行い、誤爆しないことを確認する
5. 既存の疑問文検出（「〜ですか」等、Rule 1由来）が退行していないことを1件確認する
