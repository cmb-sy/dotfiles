# meeting-notes 個人アクションアイテム化 + 廃止参照除去 設計書

## 目的・スコープ

`claude/skills-src/ops/meeting-notes/SKILL.md` を改造する。

1. アクションアイテム欄を「担当者全員分の一覧」から「自分（中島）宛のみ」に絞り込み、議事録から他人のタスクを読む際のノイズを減らす。
2. スキル起動時に議事録テキストが未提供の場合、明示的に入力を促す Step を追加する。
3. 調査の過程で、Step 3〜5 が commit `a6dbe91`（2026-05-18）で削除済みの `project-update` skill を参照し続けている dead code であることが判明したため、合わせて除去する。

対象は `meeting-notes` 単体。他 skill（slackcli, obsidian-refresh 等）への変更は行わない。

## 変更対象ファイル

- `claude/skills-src/ops/meeting-notes/SKILL.md`（唯一の変更対象。ビルド後の `claude/skills/meeting-notes/SKILL.md` は既存のビルドプロセスに従って反映する）

## 変更内容

### 1. frontmatter の description 整理

**Before**:
```yaml
description: >-
  入力された議事録テキスト（Teams 文字起こし・手書きメモ・チャットログなど形式不問）を
  構造化して保存し、議事録 + 関連 Slack チャンネルの直近メッセージを合わせて
  プロジェクトの「今の全体像」を最新化する。
```

**After**:
```yaml
description: >-
  入力された議事録テキスト（Teams 文字起こし・手書きメモ・チャットログなど形式不問）を
  構造化して保存する。アクションアイテムは自分（中島）宛のみ抽出する。
```

### 2. 冒頭説明文の整理

**Before**:
```
入力された議事録テキスト（Teams文字起こし・手書きメモ・チャットログなど形式不問）を構造化して保存し、**議事録 + 関連 Slack チャンネルの直近メッセージを合わせて**プロジェクトの「今の全体像」を最新化する。

議事録は会議1回の断面、Slack はその前後の継続議論。両方を統合することでプロジェクト状態が正しく最新化される。
```

**After**:
```
入力された議事録テキスト（Teams文字起こし・手書きメモ・チャットログなど形式不問）を構造化して保存する。アクションアイテムは自分（中島）宛の項目のみを抽出し、他の担当者宛の項目は含めない。
```

### 3. Step 0（新規）: 議事録テキストの取得

現行フローは「議事録テキストは既に渡されている」ことを前提にしている。呼び出し単体（テキスト未添付）での起動に対応するため、Step 1 の直前に新設する。

```markdown
### Step 0: 議事録テキストの取得

1. 呼び出し時に議事録テキストがまだ渡されていない場合、ユーザーに「議事録スクリプトを教えてください」と尋ねる
2. テキストを受け取るまで Step 1 には進まない
```

### 4. Step 2（旧）内、アクションアイテムのフィルタ

**Step 2 の処理手順**（「1. 入力テキストを分析し、以下の構造で整形する」の直後）に一文追加:

```
アクションアイテムは担当が「中島」と読み取れる項目のみ抽出する。それ以外の担当者宛の項目は出力に含めない。
```

**議事録フォーマットのテンプレート**内、アクションアイテム欄:

**Before**:
```
### アクションアイテム
- [ ] 内容（担当: XX, 期日: YYYY-MM-DD）
```

**After**:
```
### アクションアイテム
（担当が「中島」の項目のみ記載。他の担当者宛の項目は含めない。該当0件の場合は「中島宛の項目なし」と記載する）
- [ ] 内容（期日: YYYY-MM-DD）
```

「担当: 中島」表記は全項目共通になるため冗長として削除し、期日のみを残す。

### 5. Step 3〜5 の削除（廃止済み project-update 参照）

以下を SKILL.md から削除する:

- `### Step 3: 関連 Slack チャンネルの特定`（全文）
- `### Step 4: Slack メッセージの取得`（全文）
- `### Step 5: project-update の実行`（全文）

旧 `### Step 6: 完了報告` を `### Step 3: 完了報告` に繰り上げ、内容を以下に整理する:

**Before**:
```
- 保存した議事録ファイルのパス
- 取得した Slack チャンネルとメッセージ件数（取得した場合）
- project-update で更新されたファイルの一覧
```

**After**:
```
- 保存した議事録ファイルのパス
```

### 6. ルール節の整理

以下を削除する:

- 「議事録は事実の記録、project-update は状態の統合。役割を混同しない」
- 「Slack 取得が失敗しても議事録保存は必ず完了させる」
- 「index.md への `slack_channels` 追加時は、既存 frontmatter を壊さない」

「ユーザー確認タイミング」の項目を以下に整理する:

**Before**:
```
6. ユーザー確認タイミング: プロジェクト特定 / 同日既存ファイルの動作 / Slack チャンネル未登録時 / project-update のタスク・期日確認
```

**After**:
```
3. ユーザー確認タイミング: 議事録テキスト未提供時 / プロジェクト特定 / 同日既存ファイルの動作
```

3項目削除の結果、残るルールは「議事録ファイルは原則上書きしない」「参加者の氏名は入力に登場したもののみ記載する」「ユーザー確認タイミング」の3件のみとなり、番号を1〜3に振り直す。

## スコープ外（YAGNI）

- 自分（中島）以外のユーザー向けの汎用化（名前を可変パラメータにする等）— このリポジトリは単一ユーザー（snakashima）専用の dotfiles であり、複数ユーザー対応の需要は現時点でない
- アクションアイテム0件時の詳細な理由分類（「読み取れなかった」と「本当に0件」の区別以上の情報付与）
- Slack 連携・project-update 相当機能の代替実装 — 今回は除去のみ。再導入が必要になった場合は別途 brainstorming する

## テスト方針

`meeting-notes` は prompt-based skill であり、自動テストの対象ではない（既存 skill も同様）。手動確認で検証する:

1. サンプル議事録テキスト（中島宛アクションアイテムを含むもの）を用意し、`/meeting-notes` をテキスト添付なしで呼び出す → Step 0 で入力を促されることを確認
2. テキストを渡した後、保存された議事録 .md のアクションアイテム欄に中島宛の項目のみが記載され、他の担当者宛の項目が含まれないことを確認
3. 中島宛の項目が0件のサンプルで実行し、「中島宛の項目なし」と記載されることを確認
4. 完了報告に Slack件数・project-update関連の記述が出力されないことを確認
5. 同日に既存の議事録ファイルがある場合、上書き/追記/中止の確認が従来通り機能することを確認（既存動作の回帰チェック）
