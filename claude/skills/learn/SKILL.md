最新3件のcommitを分析し、実装で使われたすべての技術・設計判断を網羅的に抽出して学習資料を出力する。

## フロー

1. 最新3件のcommitを取得し、全差分を読む
2. commitに登場するすべての技術・ライブラリ・パターン・設計を列挙する
3. `docs/learnings/` の既存ファイルを確認し、未出力のトピックだけを対象とする
4. 各技術の公式ドキュメントリンクをWebSearchで収集する
5. ドメイン別・3層構造で学習材料を組み立てる
6. `docs/learnings/YYYY-MM-DD-<topic>.md` に出力する

## commit分析

```bash
git log --oneline -3
git show --stat HEAD~2..HEAD
git diff HEAD~3..HEAD
```

差分から読み取る対象:

- 言語・フレームワーク・ライブラリ・外部サービス・CLIツール
- 設計パターン（命名規約、抽象化の粒度、インターフェース設計）
- 削除・変更されたコードが示す「前の判断を覆した理由」
- コミットメッセージが示す意図（feat / fix / refactor の背景）

## 技術スタック抽出

commitに登場するすべての要素を漏れなく列挙する。省略しない。

| カテゴリ     | 抽出対象                                                 |
| ------------ | -------------------------------------------------------- |
| 言語・構文   | TypeScript型定義、Python型ヒント、シェル構文             |
| ライブラリ   | import/require されるパッケージ名                        |
| 外部サービス | AWS、GitHub、Slack 等のAPI・SDK・設定                    |
| 設計パターン | Observer、Factory、Strategy 等が当てはまる構造           |
| AIシステム   | プロンプト構造、ツール定義、フック設計、エージェント構成 |

このリストが出力の「技術スタック全一覧」になる。

## リンク収集

Step 2 で列挙した各技術について WebSearch で正確なURLを取得する。

```
WebSearch: "[技術名] official documentation"
WebSearch: "[技術名] [API・概念] guide"
```

取得するリンク:

- **公式ドキュメント** — リファレンス・APIガイド
- **概念解説** — 仕組みを深く説明した記事（Zenn、Medium等）
- **設計判断の根拠** — なぜその設計が選ばれるかを解説した資料
- **比較記事** — 代替技術との比較

URLはWebSearchで実在を確認してから使う。見つからない場合は `[公式ドキュメントで "{検索キーワード}" を検索]` と記載する。

## 学習材料の構成

各トピックを以下の3層で説明する。**層1だけで終わる説明は不十分。**

**層1 — 技術の仕組み（What / How）**
動作原理・内部メカニズム。「使い方」だけでなく「なぜそう動くか」まで書く。

**層2 — 設計判断とトレードオフ（Why / Why not）**
なぜこの技術・設計を選んだか。代替案との比較。AIがこの実装を選んだ理由の推論。コストとベネフィット。

**層3 — このcommitから読み取れること**
具体的なファイルパス・行番号への言及。実装の良い点と改善余地。

## 出力フォーマット

保存先: `docs/learnings/YYYY-MM-DD-<topic>.md`（`docs/learnings/` がない場合は作成する）

## 制約

- 技術スタック抽出は省略しない。commitに現れたすべての要素をリストアップする
- URLはWebSearchで実在を確認してから使う。推測で生成しない
- 層1だけで終わらず、必ず層2（Why）と層3（実装言及）を含める
- 既存の `docs/learnings/` にあるトピックは繰り返さない
- 会話内提示だけで終わらせず、必ずファイルに出力する
