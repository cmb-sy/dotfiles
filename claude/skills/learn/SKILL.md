最新5件のcommitから技術スタックと意思決定の背景を抽出し、技術・設計判断・commit内容を同時に学べる教材を生成する。

出力は「commitの記録だけ」ではなく「教材」。
読者が以下を理解できる状態をゴールとする:

- 何をしたか（What）
- なぜそれをしたか（Why）
- どういう選択肢の中でその判断になったか（Decision）
- 次に自分のプロジェクトへどう応用するか（Apply）
- commitで具体的に何が変わったか（Commit Change）

## フロー

1. 最新5件のcommit差分を取得する
2. 差分から技術要素と意思決定シグナルを抽出する
3. 学習価値の高いメイントピックを選定する
4. `docs/learnings/` の既存教材と重複しない対象に絞る
5. メイントピックごとに context7 / WebSearch で一次情報を補強する
6. commit解説 + 技術解説 + 判断背景を統合した教材として `docs/learnings/YYYY-MM-DD-<topic>.md` に出力する

## Step 1: commit差分の取得

```bash
git log --stat -5
git diff HEAD~5..HEAD
```

## Step 2: 技術と意思決定シグナルの抽出

差分に登場するすべての要素を漏れなく列挙する。省略しない。

| カテゴリ     | 抽出対象                                                 |
| ------------ | -------------------------------------------------------- |
| 言語・構文   | TypeScript型定義、Python型ヒント、シェル構文             |
| ライブラリ   | import/require されるパッケージ名                        |
| 外部サービス | AWS、GitHub、Slack 等のAPI・SDK・設定                    |
| 設計パターン | Observer、Factory、Strategy 等が当てはまる構造           |
| AIシステム   | プロンプト構造、ツール定義、フック設計、エージェント構成 |

加えて、以下を「意思決定シグナル」として抽出する:

- どの問題を解決するための変更か（バグ、保守性、速度、運用性、安全性）
- どの制約が意思決定を作ったか（互換性、期限、既存設計、チーム規約）
- 何を捨て、何を採用したか（トレードオフ）
- 削除・変更されたコードが示す「前の判断を覆した理由」
- commit message が示す意図（feat / fix / refactor の背景）
- 変更ファイルの役割分担（どのファイルで何を担当しているか）

## Step 2.5: commit内容の解説ポイント整理

各commitについて、教材に載せる解説ポイントを整理する。

- **何を変えたか**: 変更対象（機能、設定、テスト、ドキュメント）
- **なぜ変えたか**: 背景課題、トリガー、再発防止意図
- **どう変えたか**: 実装方針、差分の要点、関連ファイルの役割
- **何を守ったか**: 互換性、安全性、運用性、可読性のどれを優先したか
- **何を見送ったか**: 今回スコープ外にした案と理由

## Step 3: メイントピック選定

抽出した全技術・判断のうち、以下の基準でメイントピック（教材として深掘りする対象）を選ぶ:

- **commitで重要な役割を果たしている技術** — import1行だけの依存は除外
- **学習価値が高い技術** — 汎用性があり、他のプロジェクトでも使える知識
- **単純すぎない技術** — 標準ライブラリの基本的な使い方などは除外
- **判断の再利用価値がある変更** — 他プロジェクトでも通用する設計判断が学べる

メイントピックに選ばれなかった技術・判断は、出力の「技術スタック」表および「判断メモ」に一覧として残す。

## Step 4: 既存教材との重複排除

`docs/learnings/` の既存ファイルを確認し、同一トピックの重複出力を避ける。

- 同一技術テーマが既にある場合は、新規ファイルを作らず「既存教材の追補」として別テーマを選ぶ
- ただし、過去と異なる判断軸（例: 安全性優先から運用性優先への転換）がある場合は新規作成可

## Step 5: 情報収集

メイントピックごとに正確な情報を収集する。

**context7スキルを優先的に使う**（ライブラリの公式ドキュメントを正確に取得できる）:

```
Skill: context7 — [ライブラリ名]の[機能名]のドキュメントを取得
```

context7で十分な情報が得られない場合にWebSearchで補完する:

```
WebSearch: "[技術名] official documentation"
WebSearch: "[技術名] [概念] guide"
```

URLはWebSearchで実在を確認してから使う。見つからない場合は `[公式ドキュメントで "{検索キーワード}" を検索]` と記載する。

## 教材の構成原則

**出力は「commit解説 + 技術教材 + 意思決定教材」。**

commitは「きっかけ」であると同時に「具体例」。
教材の主役は「技術」「判断」「実際の変更内容」。
読者がこの教材だけで、実装手法だけでなく判断基準と変更の意図まで再利用できるレベルを目指す。

### 教材の深さ

commitで実際に使われた機能・概念・判断を軸に書く。技術の全体像には触れるが、深掘りはcommitに関連する範囲に限定する。

例: commitでFastAPIのデプロイ設定を変更した場合

- 書く: FastAPIの概要（2-3文）、ASGIサーバーの仕組み、uvicornの設定、本番デプロイのベストプラクティス
- 書かない: FastAPIのルーティング全解説、依存性注入の詳細、全ミドルウェアの一覧

### 悪い例と良い例

悪い例（記録止まり）:

> 積み上げ棒グラフへの変換時に、BAR_STACKED と BAR_STACKED_100 の使い分けが必要だった。

良い例（技術 + 判断教材）:

> ## 積み上げ棒グラフの2つのモード
>
> python-pptxでは積み上げ棒グラフに2つのモードがある。
>
> - **BAR_STACKED**: 実数値をそのまま積み上げる。各セグメントの絶対量を比較したいとき。
> - **BAR_STACKED_100**: 全体を100%に正規化して積み上げる。構成比率を比較したいとき。
>
> ```python
> from pptx.enum.chart import XL_CHART_TYPE
>
> # 実数値の積み上げ — 部署ごとの人数を比較
> chart_type = XL_CHART_TYPE.BAR_STACKED
>
> # 100%積み上げ — 各部署の男女比率を比較
> chart_type = XL_CHART_TYPE.BAR_STACKED_100
> ```
>
> 選択基準: 「このグラフで読者に伝えたいのは絶対量か、比率か？」
>
> 判断理由: 利害関係者が比較したい軸（量 / 構成比）に合わせると、グラフの誤読を防げる。

### 記述ルール

- **技術の概要から始める**: その技術が何で、何を解決するのかを最初に説明する
- **コード例を含める**: 概念の説明には必ず短いコード例を添える。commitのコードをそのまま貼るのではなく、学習用に簡潔にしたサンプルを書く
- **「なぜ」を説明する**: 代替手段がある場合、なぜこの技術・手法を選ぶのかを書く
- **判断テンプレートで書く**: 「課題 → 選択肢 → 採用判断 → トレードオフ → 再利用条件」を明示する
- **実践的なTipsを含める**: ハマりやすいポイント、ベストプラクティスを書く
- **自然な日本語で書く**: 技術文書として読みやすい日本語を使う。体言止めの羅列は避け、文章として成立させる

## 出力フォーマット

**1ファイル = 1つのテーマ**（関連する複数技術をまとめてよい）。

保存先: `docs/learnings/YYYY-MM-DD-<topic>.md`（`docs/learnings/` がない場合は作成する）

テンプレート（この順序で出力）:

1. `# [技術テーマ]を理解する`
2. メタ情報
   - **日付**: YYYY-MM-DD
   - **きっかけ**: [対象commitの要約]
   - **分野**: [インフラ / バックエンド / フロントエンド / AI / 開発プロセス]
3. `## 概要`
   - 技術の全体像を2-3文で説明
4. `## このcommitで何をしたか（Commit Overview）`
   - 変更の要約（2-5行）
   - 主要変更点（箇条書き）
   - 影響範囲（ファイル・機能・運用）
5. `## なぜこのcommitをしたのか（Intent & Context）`
   - 背景課題
   - 変更しない場合のリスク
   - 今回の狙い（短期 / 中長期）
6. `## 技術スタック`
   - 技術・概要・公式リンクの表
7. `## なぜこの変更が必要だったか（Problem Framing）`
   - 解決課題 / 現状の痛み / 放置リスク
8. `## 判断の分解（Decision Breakdown）`
   - 選択肢
   - 採用判断と理由
   - トレードオフ
   - 再利用条件
9. `## [技術名1]: [サブテーマ]`
   - これは何か
   - 仕組み
   - 学習用コード例（簡潔）
   - なぜこれを使うのか
   - 実践Tips
   - もっと学ぶには（公式リンク / 解説記事 / プロジェクト内実装例）
10. 必要な数だけ 9 を繰り返す
11. `## まとめ`

- 技術群の関係性と、判断基準の再利用ポイント

## 制約

- 技術スタック抽出は省略しない。commitに現れたすべての要素をリストアップする
- URLはWebSearchで実在を確認してから使う。推測で生成しない
- 「commitで〜した」という記録ではなく、技術と判断を教える教材として書く
- コード例は学習用に簡潔にする。commitの差分をそのまま貼らない
- 既存の `docs/learnings/` にあるトピックは繰り返さない
- 会話内提示だけで終わらせず、必ずファイルに出力する
- 判断理由は推測で断定しない。根拠が弱い場合は「推定」と明記する
- 教材内に最低1つは「判断テンプレート（課題→選択肢→採用→トレードオフ）」を含める
- 各教材に「このcommitで何をしたか」「なぜこのcommitをしたか」を必ず含める
