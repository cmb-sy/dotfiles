name: project-update
---
入力されたテキスト（議事録・Slackチャット・メモなど形式不問。省略可）を一次入力としつつ、**関連 Slack チャンネルと関連 GitHub リポジトリを自動走査**して情報を補完し、該当プロジェクトの状況・方針・やるべきことを整理してプロジェクトファイルを更新してください。

さらに、対象プロジェクトの **`02_projects/{PJ}/` 内トピック `.md` にユーザーが普段から書き込んでいる未整理メモ**（セクション外の自由記述・URL 羅列・会話の走り書きなど）を**一次情報として必ず読み取り**、Slack/GitHub と同列に統合したうえで、定義済みセクションへ**再配置・要約**する。

単なる記録の追記ではない。複数の情報源を**分析・統合**し、プロジェクトの「今の全体像」が一目でわかる状態に仕上げること。

## プロジェクト構成

プロジェクトは `02_projects/{PJ}/` のディレクトリ構成で管理する。

```
02_projects/{PJ}/
├── index.md          ← プロジェクト全体の概要・経緯・トピック別ファイルへのリンク
├── {トピック1}.md    ← テーマ別の詳細（状況・方針・やること・課題）
├── {トピック2}.md
└── ...
```

- **index.md**: プロジェクト全体の概要・経緯・トピック別ファイルへのリンク集。詳細は書かない
- **トピック別ファイル**: テーマごとに分離。各ファイルが独立して状況を把握できる単位にする
- 関係者はプロジェクトのコアメンバーのみ記載し、個別タスクの協力者は含めない

## 処理フロー

### Step 1: プロジェクト推定・確認

1. 入力テキスト（あれば）からプロジェクトを推定する。無ければユーザーに「どのプロジェクトを更新しますか？」と聞く
2. `02_projects/` 配下の既存プロジェクトディレクトリを一覧し、候補を提示する
3. 該当なしの場合は新規プロジェクト名を提案する
4. 既存トピックファイルに該当するか、新規トピックファイルが必要かも判断する
5. ユーザーの承認を得てから次のステップへ進む

### Step 2: 情報源の特定

対象プロジェクトの `index.md` を読み、frontmatter から関連リソースを特定する:

```yaml
slack_channels:
  - "#channel-name-1"
  - "#channel-name-2"
github_repos:
  - "owner/repo1"
  - "owner/repo2"
```

- `slack_channels` が未定義 → ユーザーに「関連 Slack チャンネルは？（複数可、skip 可）」と聞き、回答を **index.md frontmatter に追記保存**
- `github_repos` が未定義 → ユーザーに「関連 GitHub リポジトリは？（複数可、skip 可）」と聞き、回答を **index.md frontmatter に追記保存**
- 新規プロジェクトの場合も同様に初回セットアップを行う

### Step 3: Slack 走査

- **slackcli CLI を使う**（認証確認: `slackcli auth list` でワークスペースが列挙されること。`slackcli auth status` は未対応の CLI がある）
- 取得範囲:
  - 前回 index.md の `updated` 日付以降（未設定なら過去7日間）
  - 各チャンネルのメッセージ: `slackcli conversations history --channel {id} --oldest {UNIX_TS}`
  - 必要に応じて `slackcli search messages --in #{channel} --after {YYYY-MM-DD}` で補完
- ノイズ除去: リアクションのみ・絵文字スタンプのみ・短い相槌を除外
- 取得失敗時はその旨を記録して続行（プロジェクト更新は止めない）

### Step 4: GitHub 走査

- **gh CLI を使う**（認証確認: `gh auth status`）
- 取得範囲（各リポジトリ、前回 updated 以降）:
  - コミット: `gh api repos/{owner}/{repo}/commits --jq ...`（または手元に clone があれば `git log`）
  - PR: `gh pr list --repo {owner}/{repo} --state all --search "updated:>{YYYY-MM-DD}"`
  - Issue: `gh issue list --repo {owner}/{repo} --state all --search "updated:>{YYYY-MM-DD}"`
- 対象: コミット主題 / PR タイトル・説明 / Issue タイトル・コメント（本文からプロジェクト状態を読み取る）
- 取得失敗時はその旨を記録して続行

### Step 5: 統合・分析

- **ローカル・トピックファイルの未整理メモ取り込み（必須）**
  - 対象 `02_projects/{PJ}/` 配下の **`index.md` を除くすべての `.md`** を読む（`議事録/` 以下は参照用とし、必要なら最新 1〜2 件のみ要約に使う。主戦場はトピック別ファイル）。
  - 次のいずれも **Slack/GitHub と同列の一次情報**として扱う:
    - 標準セクション（`### ...現在の状況` 等）**外**にある無秩序な本文
    - フロントマター無し・見出し無しの **全文が走り書き**になっているファイル
    - URL（SharePoint / Teams / GitHub / その他）・箇条書き・会話メモの羅列
  - 取り込み後の方針:
    - **単なる末尾追記は禁止**。内容を `現在の状況` / `方針・決定事項` / `やること` / `未決・課題` / `関連情報` に**再配置**し、重複・古い推測は削除または更新する。
    - 確度が低い仮説は `未決・課題` に明示し、確定した事実だけ `方針・決定事項` に上げる。
    - 消してはいけない一次 URL は `関連情報` に残す（可能なら `[[]]` や説明1行を付与）。
    - **テンプレ非準拠ファイル**（所定の frontmatter + 5 セクションが無い）: 既存本文全体を材料に、**テンプレへ昇格**させる。昇格後、散文の重複は削る（事実は要約または `関連情報` に保持）。
- 貼り付けテキスト + **上記ローカルメモ** + Slack + GitHub を統合し、以下を抽出する:
  - **現在の状況変化**: 進捗・方針変更・ブロッカー・新しい決定事項
  - **やること候補**: 発言・コミットメッセージ・Issue・走り書きから派生する未完了アクション
  - **人物・担当**: 発言者・コミッター・アサイン・メモ内の名前から担当者を推定
  - **期日候補**: 「〜までに」「〜日」「期限」等の表現から期日を抽出
- 既存の index.md / トピックファイルの状態と差分を取り、古くなった記述は更新・削除する

### Step 6: ユーザー確認

**タスク抽出後、ユーザーに必ず質問する**（AskUserQuestion を使用）:

1. **タスク内容確認**: 抽出した「やること」候補を一覧提示し、「この内容で合っていますか？ 追加・削除・表現修正はありますか？」と聞く
   - `Other` で自由記述を受け、進捗コメント（「まだです、リマインド日だけ決めた」等）も解釈する
2. **期日確認**: 各タスクに「期日はいつですか？ 未定なら『未定』で可」と聞く（まとめて一括回答可）
3. **担当確認**: 入力から担当が読み取れない場合のみ聞く。明らかに自分の場合は省略

### Step 7: ファイル更新

- index.md のフォーマット:

```markdown
---
title: "{プロジェクト名}"
status: 進行中 | 完了 | 保留
summary: "一言で説明"
tags: [project]
updated: YYYY-MM-DD
slack_channels:
  - "#channel-name"
github_repos:
  - "owner/repo"
---

### <font color="#81A1C1">概要</font>
- **目的**:
- **背景**:
- **関係者**:
- **ゴール**:

### <font color="#81A1C1">これまでの経緯</font>
（時系列で主要イベントを箇条書き）

### <font color="#81A1C1">トピック別ファイル</font>
- [[02_projects/{PJ}/{トピック1}]] — 一行説明
- [[02_projects/{PJ}/{トピック2}]] — 一行説明
```

- トピック別ファイルのフォーマット:

```markdown
---
title: "{トピック名}"
parent: {PJ}
updated: YYYY-MM-DD
---

### <font color="#81A1C1">現在の状況</font>
（このトピックの進捗・現在地・直近の動き）

### <font color="#81A1C1">方針・決定事項</font>
（確定した方針・合意事項。理由や背景も簡潔に添える）

### <font color="#81A1C1">やること</font>
- [ ] アクション内容（担当: XX, 期日: YYYY-MM-DD）

### <font color="#81A1C1">未決・課題</font>
（結論が出ていない論点、リスク、ブロッカー）

### <font color="#81A1C1">関連情報</font>
（データソース・リンク・参考資料など）
```

- 「やること」は必ず `担当: XX, 期日: YYYY-MM-DD` の形式で統一する
- 期日が未定の場合は `期日: 未定` と明記する（空欄は禁止）
- 担当者が自分以外の場合は必ず記載する。自分の場合は `担当: 自分`

### Step 8: 完了報告

簡潔に以下を出力する:
- 使用した情報源（貼り付けテキストの有無、**トピック `.md` 未整理メモ取り込み（対象ファイル数・おおよその行数）**、Slack N件、GitHub N件の走査結果）
- 新規追加・変更・削除した index.md / トピックファイルの一覧
- 確定したタスクの件数

## ルール

1. 貼り付けテキストが無くても Slack/GitHub 走査だけで更新を実行できる
2. 新規プロジェクトは必ずディレクトリとして作成する（単体 .md ファイルにしない）
3. 既存の内容と入力内容を**統合**し、最新の状態に上書きする（古い情報は更新・削除して良い）
4. Slack/GitHub ノイズ除去: 雑談・リアクション・重複・bot通知を除外し、要点のみ抽出する
5. Obsidianのリンク記法 `[[]]` やタグ `#tag` が入力内にあれば維持する
6. Slack/GitHub 取得に失敗しても、他の情報源だけで更新は続行する（失敗は報告に明記）
7. frontmatter の `slack_channels` / `github_repos` を編集する際は既存フィールドを壊さない
8. 各セクションの内容は情報源から読み取れる範囲で埋める。情報がないセクションは空のまま残す
9. ユーザー確認のタイミング:
   - プロジェクト特定（Step 1）
   - Slack/GitHub 情報源の初回登録（Step 2）
   - タスク内容・期日・担当（Step 6）
10. トピック `.md` への**随時の走り書きメモは奨励**とみなす。`/project-update` 実行時に Slack/GitHub とあわせて読み取り、**セクション構造へ再編成**して読みやすく戻す（インボックス → 整形式）。
