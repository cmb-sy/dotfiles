やったこと・判断したことを問題形式で振り返り、理解のギャップを可視化する。
**いつでも・何に対しても**実行できる汎用スキル。

**vault パス:** 本スキルの vault 参照はすべて `$HOME/develop/obsidian` 起点の絶対パスで解決する（どのリポジトリの cwd から呼ばれても動く）。vault が存在しない場合は過去問.md への記録・北極星参照をスキップし、出題と解説のみ行う（クイズ自体は成立させる）。

**モード:**
- **引数なし**: 本日のデイリーファイル + セッションログから自動収集して出題
- **テキスト引数**: 渡した説明をそのまま素材として出題（ファイル不要）
  - 例: `/generate-problem "Databricks で VPC Peering を設定し、外部サービスとの接続を確立した"`
- **日付引数**: 指定日のデイリーファイルを素材として出題
  - 例: `/generate-problem 2026-04-24`
- **learn 連携**: learn skill が教材生成直後にテキスト引数モードで呼ぶ（素材 = 教材の要約 + 用語リスト）。挙動はテキスト引数モードと同一
- **`--quick`**: 問題数は **5問のまま**変えない。受入検証 NG 時の修正再依頼だけスキップして速度を稼ぐ（作業直後の素早い確認用）

## 処理フロー

### Step 1: 素材収集

引数の種類に応じて素材を決定する。

**テキスト引数あり:**
- 渡されたテキストをそのまま素材とする
- デイリーファイル・セッションログは読まない
- 「今やったことの内容」として扱う

**日付引数 or 引数なし（本日）:**
- 以下を並列取得:
  - デイリーファイルの `## 今日の成果` セクション（`<!-- BEGIN:accomplishments -->` ブロック + マーカー外の手動追記）
  - `<!-- BEGIN:session-digest -->` 〜 `<!-- END:session-digest -->`
  - なければ `~/.claude-work/projects/` から当日 JSONL（最新ファイル優先）
- `$HOME/develop/obsidian/01_quant/北極星.md` を読み込む（意義軸の問題生成に使う）

**素材が何もない場合:**
- 「何について振り返りますか？今やったことを教えてください」とユーザーに聞く
- 回答をそのまま素材として使う

**過去に扱った技術の収集（技術問題プール / 全モード共通）:**

技術問題は「今日やったこと」だけでなく、**過去に扱ってきた技術全体**から出題する。以下を収集して技術キーワードのプールを作る:

- 直近のデイリーファイル群（`$HOME/develop/obsidian/00_daily/` 配下、当月＋前月を目安）の `session-digest` / `今日の成果` から技術キーワード・ツール名・概念を抽出する
- `$HOME/develop/obsidian/01_quant/過去問.md` を読む。用途は2つ: (1) **既出問題の重複回避**（同じ問いを再利用しない。問題のコピー禁止）、(2) **復習期日判定**（`次回出題日` が今日以前の行の抽出。「復習枠」参照）
- セッションログ（`~/.claude-work/projects/`）に技術的文脈があれば補強に使う

抽出した技術キーワードをプールとして保持し、Step 2 の技術問題生成で使う。本日扱った技術に偏らせず、過去のトピックも織り交ぜる。

---

### Step 2: 問題生成

**問題数は 5 問固定**。素材が薄い日も豊富な日も同じ 5 問にする（量で揺らさない）。`--quick` 時も 5 問のまま — 速度差は受入検証 NG 時の修正再依頼をスキップすることで稼ぐ。

**軸の配分（5問固定・厳守）:**

| 軸 | 問題数 | 出題の観点 |
|---|---|---|
| **技術的理解** | **3問** | 仕組み・原理・落とし穴・ベストプラクティス。なぜそれが正しいか／次に直面する問題は何か |
| **意義・北極星接続** | **1問** | この作業は北極星（長期目標）にどう効くか。実績として説明できるか |
| **判断根拠 or リスク・次の一手** | **1問** | なぜAでなくBを選んだか／この判断で捨てたものは何か／次に同じ状況になったら何を変えるか。判断軸でいずれか1問を選ぶ |

合計 5問。3 + 1 + 1 の配分を揺らさない。

**復習枠（間隔反復 / spaced repetition）:**

1. 出題前に `date +%F` を実行して今日の日付を確定する（LLM の日付推測禁止）
2. `$HOME/develop/obsidian/01_quant/過去問.md` に `次回出題日` が今日以前の行があれば、技術3問のうち1問をその復習に充てる（最も古い期日の行を優先。5問固定・3+1+1 配分は不変）
3. 復習は**変形出題**とする: 元の行と同じ概念を、別の文面・別の選択肢・可能なら別の角度で問う（丸暗記防止。元の問題文の再利用禁止）
4. 復習結果による行更新は Step 5 で行う:
   - 正解: `復習回数` を +1、`次回出題日` を間隔列 [1日, 3日, 7日, 30日] の次へ進める。30日間隔で正解したら卒業（`次回出題日` を `—` にする）
   - 不正解・わからない: `復習回数` を 0 に戻し、`次回出題日` を翌日にする
5. 復習期日の行がなければ復習枠は使わず、通常の技術問題3問とする
6. 旧形式の行（`次回出題日` 列がない行）は復習対象にしない（読み取り互換のまま触らない）

**ハーネス枠（learn 連携時のみ）:**

素材にハーネス文脈（このプロジェクトの現在の構築・次の構築ステップ。learn の「ハーネス観点」節が供給する）が含まれる場合、技術3問のうち1問を**ハーネス問**にする:

- 問うのは**構築の理解**: 「このプロジェクトのハーネスは今どう組まれているか（何がどこで何を防いでいるか）」「次に何をどこへどう構築すべきか、なぜそれが先か」
- スコア・成熟度 Level の数値当てクイズにしない（素材に数値が含まれていても問いの対象にしない）
- 誤答選択肢は「ありがちな構築の誤り」（過剰な仕組み・構築順序の誤り・置き場所の誤り）から作る

**技術3問の枠割当（優先順位）:**

1. 復習枠（期日到来行があるとき）
2. ハーネス枠（ハーネス文脈があるとき）
3. 通常問（残り）

両方発動時は 復習1 + ハーネス1 + 通常1。**予測型1問の要件は復習問・ハーネス問で兼ねてよい**（例: 「この構築を先送りしたら何が起きるか」は予測型かつハーネス問）。

**難易度のバリエーション（5問固定・厳守）:**

5問の難易度配分は **初級 1・中級 2・上級 2**。これを必ず守る。
- `初級`: 用語の定義・基本動作の理解（「そもそも何か」を確実にする）
- `中級`: 仕組み・使い分け・典型的な落とし穴
- `上級`: 内部実装・エッジケース・パフォーマンス/セキュリティ・設計トレードオフ

**技術問題の生成（Claude Code 自身が考える / Web検索必須 / 過去問の再利用禁止）:**

技術問題は「今日やったことの確認」に留めず、**過去に扱ってきた技術から Claude Code 自身が新規に考えて出題する**。

1. Step 1 で集めた技術キーワードプール（過去のデイリー＋本日の素材）から、出題する技術トピックを選ぶ。本日扱った技術に偏らせず、過去のトピックも織り交ぜる
2. 選んだトピックを `WebSearch` で検索し、**最新の公式ドキュメント・ベストプラクティス・よくある落とし穴・新しい仕様**を調べる。これを問題の素材にする
3. 検索で得た知識をもとに、**全く新規の問題**を作る:
   - `$HOME/develop/obsidian/01_quant/過去問.md` の既出問題と内容が被らないこと（同じ技術でも別の角度・別の論点にする）
   - 「やったことの再確認」ではなく「その技術の原理・限界・応用」を問う
   - 誤答選択肢は実際の誤解・落とし穴から作る（ランダムな出鱈目にしない）
   - 技術3問のうち**最低1問は予測型**にする: 「この変更を加えたら何が起きるか」「この設計を別の文脈に移すとどこが壊れるか」「この前提が崩れたら何が起きるか」を問う（主に中級・上級に置く。理解とは次の一手を出せる状態であり、定義暗記ではない）
4. 参照した URL を問題ごとに記録する（Step 3 の解説と Step 5 の書き込みで使う）

**正解位置の完全ランダム化（必須）:**

- 問題を組み立てる**前に、5問分の正解位置を機械的な乱数で決める**（例: `[B, C, A, B, A]`）。決めた配列を後から内容に合わせて入れ替えない
- 配分の偏りを意図的に補正しない — A 連続も B 偏りも、乱数が出した結果はそのまま採用する（人為的調整＝バイアスの再混入）
- `わからない` は常に **D（最後）に固定**。「思考停止用」ではなく「ギャップを可視化する正規の選択肢」なので毎問必ず含める
- LLM は無意識に「最初の選択肢を主役（=正解）にしがち」というバイアスがある。**完全ランダム化はこのバイアスを潰すための装置**であって、見た目の体裁ではない

**問題文のフォーマット（長文形式）:**

各問題は以下の構造で作成する。質問文は背景・状況説明を含む長文にすること。**問いや選択肢で使う専門用語・プロジェクト固有の概念（例: 保証点・受入検証・done-criteria）は、背景の中で1行定義してから使う** — 読者はその用語を初見と仮定する。定義なしのジャーゴンで問うと、知識でなく「文脈を覚えているか」のテストに化ける。

```
[軸 / 難易度 / N問目/全M問]

背景: {問いの前提となる状況・技術的文脈を2〜3文で説明する。問い・選択肢に登場する
      専門用語はここで「◯◯とは〜のこと」と1行定義する。技術問題では、
      本日の作業に限らず「その技術が使われる典型的な場面・前提」を述べてよい。
      判断・意義軸では「なぜその選択をしたか」「どういう制約があったか」を含める。}

問: {背景を踏まえた本文の問い。「なぜ」「どうすべきか」「何が起きるか」など
    判断・理解・予測を求める形にする。}

A. {選択肢（具体的な技術用語・手順・数値を含む）}
B. {選択肢（ありがちな誤解・落とし穴を反映）}
C. {選択肢（別の視点・代替手段）}
D. わからない
```

※ 正解は A に固定せず、上記ランダム化ルールに従って A/B/C のいずれかに置く。

**並列生成（サブエージェント方式）:**

生成は問題ごとに独立なので、サブエージェントで並列化する。ユーザーが1問目に回答している間に残りが裏で完成する構造にし、体感待ち時間を「1問分の生成時間」まで縮める。

1. **割当表の確定（オーケストレーター、dispatch 前に必ず行う）**: 5問分について「軸 / 難易度 / トピック / 枠種別（復習/ハーネス/通常）/ 予測型フラグ（技術3問のどれを予測型にするか。復習問・ハーネス問が兼ねてよい）/ 正解位置（上記ランダム化ルールで事前決定）/ 重複禁止リスト（過去問.md の既出要旨）」を確定する。割当なしの dispatch を禁止する（並列生成でトピック重複・正解位置バイアスが復活するため）
2. **並列 dispatch**: 1問につき生成サブエージェント1体を background で起動する。プロンプトは自己完結にする: 素材の該当部分・割当（軸/難易度/トピック/枠種別/予測型フラグ/正解位置）・問題文フォーマット・選択肢構成ルール（正解1+誤解1+代替1+D固定）・解説3点セット（用語解説/解説/Tips、各8行以上・初学者前提・なぜ→何を→どうやって順）・参照 URL の記録・復習枠の場合は元の過去問行の内容と変形指示（同じ概念を別の文面・別の選択肢で問う。元の問題文の再利用禁止）、をすべて含める。各エージェントは WebSearch で裏取りし、構造化された1問（問題文・選択肢・正解・用語解説・解説・Tips・参照URL・出題理由）を返す
3. **生成エージェントの自己チェック（返却前）**: 背景が具体的か / 誤答が実際の誤解・落とし穴か / 解説が要件を満たすか / 割当どおりか
4. **受入検証（オーケストレーター、各問の到着時）**: 割当遵守（軸・難易度・枠種別・予測型フラグ・正解位置）/ 選択肢4つ（D=わからない）/ 既出非重複（コピーか否かで判定。復習枠は変形出題のため概念一致は許容）/ セット全体で北極星1問の充足（予測型は割当表で構造的に保証済み）、を機械的に確認する。NG は該当エージェントに1回だけ修正を再依頼する（`--quick` 時は再依頼せず受け入れて先へ進む）
5. **出題開始**: 割当表の1問目が受入を通り次第 Step 3 の出題を開始する。全問の完成を待たない。出題は割当表の順で行い、次の問題が未着ならその到着を待ってから出題する。以降はユーザーの回答と並行して残りが到着する。生成エージェントが失敗・長時間未着（目安3分超）の場合は待ち続けず、オーケストレーターがその問を割当どおりその場で inline 生成して続行する
6. **追撃・採点は inline**: 追撃問題はユーザーの誤答内容に依存するため事前生成できない。オーケストレーターがその場で作る

**その他の問題生成ルール:**
- 選択肢は **A/B/C の本物 3 つ + D の `わからない`、合計 4 つに固定**。これより増やさない、減らさない
- 正解は1つ（`multiSelect: false`）
- 本物の選択肢 3 つは「正解 1 + ありがちな誤解 1 + 別視点の代替 1」で構成する（ランダムな出鱈目を入れない）
- テキスト引数モードでも北極星を自動で参照し、意義軸を必ず1問含める
- `--quick` 時も 5問・3+1+1 配分・初級1中級2上級2 を守る。スキップするのは受入検証 NG 時の修正再依頼のみ
- 各問題には**出題理由・難易度・参照 URL**を内部に記録し、Step 5 のデイリー書き込みに含める

### Step 3: 順次出題

`AskUserQuestion` で1問ずつ出題する。選択肢は**参考**であり、自由記述（Other）が常に最良の回答手段。

**ダイアログの自己完結:** AskUserQuestion の question フィールドには要旨1行ではなく、**背景の要約（用語定義を含む）+ 問い**を入れる。チャット本文を読み返さなくてもダイアログ単体で答えられる状態にする（本文の背景とダイアログが分離すると、ダイアログしか見ていない回答者には用語が初見になる）。

**回答の受け取り方:**
- **選択肢を選んだ場合**: 選択内容を回答として評価する
- **Other（自由記述）が来た場合**: 記述内容を正解基準と照合し、自分の言葉で評価する。部分正解も認める
- **わからない**: 「次へ進みます」と返し、解説のみ提示する（正解扱いしない）

**回答直後の処理（各問ごと）:**

色分けルール（セクションごとに以下の見出しで明示する）:
- 判定行: 「正解です。」/ 「正しくは○○です。」/ 「正解は○○です。」
- `[用語解説]` — 青系ラベルで表示
- `[解説]` — 緑系ラベルで表示
- `[Tips]` — 紫系ラベルで表示

**解説の前提（重要）:**

**読み手はその分野の初学者だと仮定して書く**。「これくらいは知っているはず」を一切捨てる。プロ向けに圧縮された定義文ではなく、初めてこの分野に触れる人が腑に落ちるまで展開する。専門用語をジャーゴンのまま並べたら失格。

フォーマット:

```
{正解 / 不正解の判定文}

[用語解説]
{**初学者がこの分野を初めて聞く前提**で書く。行数の目安は 8 行以上（5 行では足りない）。
 以下をすべて満たすこと:

 - **初出定義の併記**: 専門用語の初出には日本語訳・英語表記・読み方を併記
   例: 「冪等性（idempotency、べきとうせい）とは〜」「reflow（リフロー / 再レイアウト計算）とは〜」
 - **「そもそも何か」を一度ほぐす**: 用語をいきなり使わず「これは要するに〜」「平たく言えば〜」で日常語に変換してから本題へ
 - **比喩・例え**: 抽象概念は「例えば〜のようなもの」で日常感覚に接地する（料理・郵便・本棚など身近な比喩）
 - **対比で説明**: A と B の対立軸があれば「A はこういうもの／B はこういうもの／違いは〜」で並べる
 - **内部でどう動くか**: 名前を出すだけでなく「内部で何が起きているか」を具体的な処理として述べる
 - 公式ドキュメント・一般的な定義と照合して正確に書く（嘘を平易に書かない）}

[解説]
{**初学者が「なぜそうなるのか」を腑に落とせる順序**で書く。行数の目安は 8 行以上。
 構成は **「なぜ → 何を → どうやって」の順序を厳守**:

 - **なぜ正解か（原則・力学から先に）**: 結論を最初に置かず、その問いの背後にある原則を先に説明する
 - **何が起きるか**: 正解を選んだら何が成立するか／選ばなかったら何が壊れるか
 - **どうやって判断するか**: 同じ状況に次回直面したら、どう考えればよいか（判断のフレームワーク）
 - **なぜ他の選択肢が誤りか**: A・B・C それぞれについて独立に説明する（「正解じゃないから」では足りない。
   どこが惜しいか、どんな誤解が背後にあるかまで）
 - **今日の作業との接続**: ユーザーが今日触れた具体的なファイル・コード・判断に引き付ける（抽象論で終わらせない）}

[Tips]
{**初学者が現場で即座に使える形**に整える。行数の目安は 8 行以上。

 - **周辺知識**: 問題文には出なかったが知っておくと役立つこと（関連 API、公式の推奨、業界慣習）
 - **具体性の最低基準**: 「速い」「多い」では足りない。「16.7ms（60fps の 1 フレーム）」「100 行」のように
   数値・コマンド・設定値・行数を必ず添える
 - **落とし穴**: 「やってしまいがちな失敗 → なぜダメか → 代わりにどうするか」の 3 点セットで書く
 - **次のステップ**: 今回理解したことの一歩先（モダンな代替、関連 API、公式の推奨パス）を最後に添える
 - **確認コマンド・確認方法**: 自分で再現できる手順を1つは入れる
   例: 「DevTools の Performance タブで Long Task を確認」「`aws s3 ls s3://bucket --summarize` で残オブジェクト確認」}
```

- **正解 / 概ね正解**: 「正解です。」から始める
- **不正解・部分正解**: 「正しくは○○です。」から始める
- **わからない**: 「正解は○○です。」から始める
- 回答・コメント・ユーザーの記述を全問分蓄積する（Step 4・Step 5 で使う）

**不正解時の追撃1問（--quick 時はスキップ）:**

- 不正解・わからないの直後、解説を提示した後に、同一概念を**別角度**から問う軽い確認問題を1問だけ即時出題する（その場で理解ギャップを閉じる）
- 追撃問題も A/B/C + D（わからない）の4択・同フォーマットの解説付き。難易度は元の問題より1段下げてよい
- 上限は1セッション **+2問** まで（3問目以降の不正解には追撃しない）。追撃への追撃はしない
- 追撃問題の結果も通常問題と同様に Step 5 で過去問.md へ記録する（間隔反復の対象になる）

**中断してよい（長引いたとき）:**

- 出題・追撃・明確化のやり取りが長引いた兆候（追撃が連鎖する / 同一問題で明確化が繰り返される / ユーザーが疲れ・中断を示唆する）が出たら、全5問+追撃の完走にこだわらず**その場で締めてよい**
- 中断時は、そこまでの正解率サマリ（Step 5 のサマリ形式）を出し、未出題分はスキップとして扱う。回答済みの不正解・わからないは通常どおり過去問.md へ記録する（間隔反復は途中中断でも成立させる）
- これは「毎回短くする」ではなく「長引いたら中断を選べる」という許可。短く進むセッションは通常どおり完走してよい

`--quick` 時は Step 4 をスキップして Step 5 へ。

**過去問.md への書き込み（毎回必須）:**

`$HOME/develop/obsidian/01_quant/過去問.md` に**不正解・わからない問題のみ**を**1行1問の表**で追記する。ファイルがなければ以下のヘッダーで新規作成する。

```markdown
---
title: "間違えた問題ログ"
updated: {YYYY-MM-DD}
---

# 間違えた問題ログ

不正解・わからない問題のみを蓄積する復習帳。新しい日付ほど下に追記する。

| 日付 | 軸 | 難易度 | 問題の要旨 | 正解 | 自分の回答 | 用語解説 | 解説 | Tips | 次回出題日 | 復習回数 |
|---|---|---|---|---|---|---|---|---|---|---|
```

各不正解・わからない問題は、表の末尾に **1行**追加する。用語解説・解説・Tips は別ブロックに分けず、**すべてセル内に収める**:

```markdown
| {YYYY-MM-DD} | {軸} | {初級/中級/上級} | {問題の要旨20字以内} | {正解の内容（記号でなく内容で記す）} | {自分の回答 or "わからない"} | {用語解説5行以上} | {解説5行以上。なぜ正解か・他選択肢が誤りな理由・判断フレームワークを含む} | {Tips5行以上。具体的なコマンド・数値・落とし穴を含む} | {翌日の日付 YYYY-MM-DD} | 0 |
```

**セル内記法（表崩れ防止）:**
- 改行は入れず1行で書く（読点・句点で区切って連続させる）。どうしても改行が要る場合のみ `<br>` を使う
- セル内のパイプ `|` は `\|` にエスケープする
- HTML コメント `<!-- -->` を含める場合はバッククォートで囲む
- 用語解説・解説・Tips は**情報量を落とさず**1セルに詰める（5行相当の内容を保持する）

**書き込みルール:**
- 正解した問題は記録しない（例外: 復習枠で出題した行の更新は正解・不正解を問わず行う）
- 新規の不正解行: `次回出題日` = `date +%F` の翌日、`復習回数` = 0
- 復習枠で出題した既存行の更新: 正解なら `復習回数` +1・`次回出題日` を次の間隔（1日→3日→7日→30日。30日で正解なら `—`）へ、不正解なら `復習回数` 0・`次回出題日` 翌日
- 旧形式の行（11列未満）は変換せず放置する（復習対象外。読み取り互換維持）
- `--quick` 時も同様に不正解のみ記録する
- 用語解説・解説・Tips は **Step 3 の回答直後出力 / Step 5 のデイリー書き込みと同じ内容**を入れる（短縮しない）
- 正解列は選択肢の記号（A/B/C）でなく**内容**を記す（正解位置はランダム化されるため記号は無意味）
- デイリーと過去問の両方に同じ内容を残すことで、過去問.md は「間違えた問題だけを集めた復習帳」として独立して読める状態を保つ
- 新しい日付が表の下になるよう追記する

## ルール

1. **素材がなくても中止しない** — ユーザーに「今やったことを教えて」と聞いて素材を得る
2. **技術問題を主軸にする** — 全体の過半数を技術的理解の問題にする。意義・判断・リスクは残りで補う
3. **技術問題は Claude Code 自身が新規に考える** — 過去に扱った技術を `WebSearch` で調べ、原理・限界・応用を問う新規問題を作る。過去問.md の既出問題を再利用・焼き直ししない（同じ技術でも別の角度にする）
4. **難易度を混在させる** — 初級・中級・上級を必ず混ぜる。全問同一難易度に偏らせない
5. **正解位置をランダムにする** — 正解を A に固定しない。A/B/C に分散させ、同位置の連続を避ける。`わからない` は常に最後
6. **不正解でも責めない** — ギャップの可視化が目的
7. **北極星接続の問いは毎回必ず1問含める**（`$HOME/develop/obsidian/01_quant/北極星.md` を参照）
8. **デイリーファイルが存在しない場合**: パスを提示してユーザーに作成を促す。作成されたら書き込む
9. **テキスト引数は短くても動かす** — 1文でも素材として十分
10. **--quick 時は修正再依頼をスキップ** — 受入検証 NG でも再生成せず受け入れて進む。問題数（5問）・配分（3+1+1）・難易度（初級1中級2上級2）は通常時と同じ
11. **用語解説・解説・Tips は必ず8行以上書く** — 初学者前提で平易に展開すると 5 行では収まらない。行数は丁寧さの最低指標であり上限ではない
12. **専門用語の初出は必ず併記**（日本語訳・英語表記・読み方）。「そもそも何か」を一度ほぐしてから本題に入る — ジャーゴンをジャーゴンのまま並べない
13. **解説は「なぜ → 何を → どうやって」の順序を厳守** — 結論先出しではなく、原則・力学から書く
14. **問題数は 5 問固定、選択肢は A/B/C/D の 4 つ固定**（D は常に「わからない」）。配分・難易度・選択肢数を量で揺らさない
15. **技術3問のうち最低1問は予測型** — 挙動予測・転移・前提崩しを問う（定義暗記に留めない）
16. **復習枠は技術3問内の1枠、追撃は最大+2問** — 5問固定・3+1+1 配分は復習・追撃で揺らさない（セッション上限7問）
17. **日付は必ず `date +%F` で確定する** — 次回出題日の計算・期日判定に LLM の日付推測を使わない
18. **並列生成は割当表なしで dispatch しない** — 軸・難易度・トピック・正解位置・重複禁止リストをオーケストレーターが先に確定して各生成エージェントへ配る（並列化でバイアス・重複が復活するのを防ぐ）
19. **ハーネス問は構築理解を問う** — 「今どう組まれているか」「次に何をどこへ構築すべきか」。スコア・Level の数値当てにしない
